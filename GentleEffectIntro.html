A Gentle Introduction
to Effect Systems

What side effects are, why they matter,
and how a compiler can track them

What Is a Side Effect?

Self-contained — no side effects

def add(x: Int32, y: Int32): Int32 =
    x + y

Takes two numbers, returns their sum. Nothing else in the program changes.

Has a side effect

def greetAndAdd(x: Int32, y: Int32): Int32 \ IO =
    println("Adding ${x} and ${y}");
    x + y

Does the same arithmetic, but also prints to the screen — an interaction with the world outside the function.

The \ IO is how Flix marks that fact in the type.

Common Side Effects

A side effect is anything a function does beyond returning a value:

Side EffectExample
I/OPrinting to the screen, reading from the keyboard
File systemReading or writing files
NetworkFetching a URL, sending a message
RandomnessRolling a dice, generating a UUID
ExceptionsThrowing an error that unwinds the call stack
Mutable stateModifying a global variable or shared data structure

The Problem with Invisible Effects

Most type signatures tell you what goes in and what comes out. They say nothing about what the function does to the world:

// Does this write to a database? Send an email? Throw? You cannot tell.
func processOrder(order Order) Result

This has real consequences:

  • Testing is hard. You need to set up every external system the function might touch.
  • Reasoning is hard. When a bug appears, you cannot tell which functions caused it without reading every body.
  • Order matters in surprising ways. Two calls that look independent may share hidden state.
  • Refactoring is risky. Moving a call can silently change behaviour.

Pure Functions: The Gold Standard

Given the same inputs, a pure function always returns the same output — and does nothing else.

add(3, 4) is always 7, no matter when or how many times you call it. Nothing else happens.

Because pure functions are predictable:

  • Testing is trivial. Call with inputs, check output. No setup needed.
  • Results can be cached. Same input always gives the same answer.
  • Safe to parallelise. Two pure functions can never interfere with each other.
  • Calls can be reordered. No hidden ordering dependencies.

The challenge: most languages have no way to guarantee a function is pure. Nothing stops someone from accidentally adding a println inside it.

Making Effects Visible

An effect system records effects in the type of a function. The compiler reads and enforces them.

def greet(name: String): Unit \ IO =
    println("Hello ${name}!")

──────────────────────  ──────  ─────
 parameter               return  effect
                         type    annotation

The \ IO is not a comment — it is part of the type. Omit it when the function is impure and the compiler rejects your code:

// Does not compile — println requires IO effect
def silentGreet(name: String): Unit =
    println("Hello ${name}!")

If a function has no effect annotation, the compiler has verified it performs no side effects. Purity is a proven fact, not a convention.

The Effect Set

Effects form a set. A function can have zero, one, or many:

// No effects (pure) — annotation omitted
def inc(x: Int32): Int32 = x + 1

// One effect
def logValue(x: Int32): Unit \ IO =
    println("Value: ${x}")

// Multiple effects — curly braces denote a set
def fetchAndLog(url: String): String \ {IO, Net} =
    let response = Http.get(url);
    println("Got response");
    response

Think of the effect set as a list of permissions. An empty set means pure:

def inc(x: Int32): Int32 \ { } = x + 1   // explicitly pure
def inc(x: Int32): Int32 = x + 1          // same — preferred

Effects Are Contagious (In a Good Way)

If you call an impure function, your function becomes impure too. Effects propagate upward through the call chain:

main()              -- calls sendReport → \ IO
  └─ sendReport()   -- calls logToFile  → \ IO
       ├─ formatReport()   -- pure, no effects
       └─ logToFile()      -- writes a file   → \ IO

This is not a burden — it is the feature. The signatures tell the whole story without reading any body:

def processData(data: List[Int32]): List[Int32]        // purely transforms data
def processAndSave(data: List[Int32]): Unit \ IO        // also writes somewhere

Effect Polymorphism

Should List.map be pure or impure? It depends on the function you pass.

// pure — just adds 1 to each element
List.map(x -> x + 1, myList)

// impure — prints each element
List.map(x -> { println(x); x }, myList)

Flix handles this with an effect variable ef in the signature:

def map(f: a -> b \ ef, l: List[a]): List[b] \ ef

"map has whatever effect f has." Filled in automatically at each call site:

Function passedEffect of fEffect of map
x -> x + 1{ } (pure){ } (pure)
x -> { println(x); x }IOIO

One implementation of map works correctly for both cases. No pure/impure variants needed.

User-Defined Effects

Flix lets you define your own effects. The key insight:

Define what an effect is separately from how it is handled.

Declare the effect

eff Ask {
    def ask(prompt: String): String
}

No implementation here — just the declaration that this capability exists.

Use the effect

def greetUser(): Unit \ {Ask, IO} =
    let name = Ask.ask("What is your name?");
    println("Hello, ${name}!")

greetUser does not know whether Ask.ask reads the terminal, returns a test value, or reads a config. That decision is made by a handler.

Handlers: Giving Effects Their Meaning

def main(): Unit \ IO =
    run {
        greetUser()
    } with handler Ask {
        def ask(prompt, resume) =
            println(prompt);
            let answer = readLine();
            resume(answer)
    }

What happens at runtime

  1. greetUser calls Ask.ask("What is your name?")
  2. The handler intercepts the call
  3. The handler prints the prompt
  4. The handler reads input — say, "Alice"
  5. The handler calls resume("Alice")
  6. greetUser continues with name = "Alice"
  7. greetUser prints "Hello, Alice!"

After the handler block, the Ask effect is gone. main has \ IO only.

Why Separate Effects from Handlers?

greetUser is written once. Swap the handler to change the behaviour:

Production — reads from terminal

run { greetUser() }
  with handler Ask {
    def ask(prompt, resume) =
        println(prompt);
        resume(readLine())
}

Tests — no terminal needed

run { greetUser() }
  with handler Ask {
    def ask(_prompt, resume) =
        resume("Alice")
}

This is dependency injection, enforced by the type system — without any framework, interface, or annotation beyond the effect declaration itself.

Non-Resumable Effects: Exceptions

A Void return type means the handler cannot resume — like an exception:

eff DivByZero {
    def divByZero(): Void
}

def divide(x: Int32, y: Int32):
    Int32 \ DivByZero =
    if (y == 0) DivByZero.divByZero()
    else x / y
def main(): Unit \ IO =
    run {
        println(divide(10, 2));  // 5
        println(divide(10, 0))   // raises
    } with handler DivByZero {
        def divByZero(_resume) =
            println("Error: div by zero")
    }

Void has no values, so _resume cannot be called. When divide(10, 0) raises, the handler intercepts and the second println never runs. This is a checked exception — composable, first-class, tracked by the type system.

Why This Matters

"What does this function do to the world?"

In most languages that question has no reliable answer without reading the implementation. In Flix, the answer is always in the type signature — and the compiler enforces it.

  • Navigation is easier. Scan signatures to find what touches the outside world.
  • Testing is simpler. Pure functions need no mocking, no test doubles, no setup.
  • Refactoring is safer. The compiler tells you when you accidentally add an effect.
  • Logic stays reusable. Write greetUser once; swap the handler for production, testing, or any other context.

Effect systems require you to think about what your functions do and say so in their types. That discipline pays off quickly — code that was previously a guess becomes code you can reason about with confidence.
