A Gentle Introduction
to Delayed Execution in Flix

Thunks, lazy values, and how effect handlers
suspend the world

What Does "Delay" Mean?

Think about a recipe. When you read a recipe, no cooking happens. The instructions sit on the page, patient and inert. The meal only exists when someone decides to follow them.

Most code works the opposite way — it is eager. The moment the interpreter reaches an expression, it evaluates it:

let x = 2 + 3   // the addition runs immediately — x is already 5

That gap between describing a computation and running it is what we mean by delay.

Flix offers three mechanisms for this, each more powerful than the last:

  • Thunks — a plain function that holds an expression until called
  • Lazy[T] — built-in lazy evaluation with memoization
  • Effect handlers — suspend the rest of the program itself

Thunks: The Simplest Delay

A thunk is a function that takes no arguments. Its type is Unit -> a — a sealed envelope containing instructions that run only when you open it.

Eager — runs immediately

let x: Int32 = 2 + 3
// x is already 5

Thunk — runs when called

let t: Unit -> Int32 = () -> 2 + 3
// nothing computed yet
let y: Int32 = t()
// now it runs

The only difference is the () -> wrapper. Without it, Flix evaluates immediately and stores the result. With it, Flix stores the recipe and waits.

Thunks are universal — this pattern works in every language with first-class functions.

Why Thunks Matter: Conditional Logging

A debug logger should only do expensive work when debugging is actually enabled. Without a thunk, the message is always built — even when never printed.

Naive — eager argument

def debugLog(on: Bool, msg: String): Unit \ IO =
    if (on) println(msg) else ()

// The expensive work runs even when on = false!
debugLog(false, "Result: ${expensiveWork()}")

Thunk — deferred argument

def debugLog(on: Bool, make: Unit -> String \ IO):
    Unit \ IO =
    if (on) println(make()) else ()

// The expensive work only runs if on = true
debugLog(false, () -> "Result: ${expensiveWork()}")

Downside: if you call a thunk twice, it runs twice — no memory of the previous result. That is what Lazy[T] fixes.

lazy and force: Built-In Lazy Evaluation

let x: Lazy[Int32] = lazy (1 + 2)   // not evaluated yet
let result: Int32 = force x          // evaluates now — result is 3
let again:  Int32 = force x          // returns 3 instantly — no recalculation

lazy (expr) wraps the expression without evaluating it. force x evaluates it — once — and caches the result.

Three properties:

  • Delayed. The expression inside lazy (...) does not run until you force it.
  • Memoized. Force it ten times — the expression runs exactly once. The result is cached.
  • Pure only. The expression inside lazy must have no side effects. The compiler enforces this.
// Does NOT compile — intentionally wrong
let bad: Lazy[Unit] = lazy (println("hi"))
// Error: expected pure expression, but expression has effect: IO

Thunk vs. Lazy[T]

When to use which:

Thunk () -> aLazy[T]
Evaluate later?YesYes
Evaluates once?No — runs each callYes — memoized
Can have effects?YesNo — pure only

Example: an expensive pure computation needed in several places.

Thunk — runs three times

let t = () -> List.range(0, 1_000_000) |> List.sum
let a = t()   // full work
let b = t()   // full work again
let c = t()   // full work again

Lazy — runs once

let lz = lazy (List.range(0, 1_000_000) |> List.sum)
let a = force lz   // runs once
let b = force lz   // cached
let c = force lz   // cached

Lazy Data Structures: DelayList

A normal list node holds an already-computed tail. A lazy list node holds a not-yet-computed tail — wrapped in Lazy:

// Conceptual shape (illustrative — not real code)
enum DelayList[a] {
    case DNil
    case DCons(a, Lazy[DelayList[a]])
}

Push this idea to its limit: a list that is conceptually infinite, computing only as much as you request.

def main(): Unit \ IO =
    let naturals = DelayList.from(1)                     // 1, 2, 3, ... forever
    let evens    = DelayList.map(x -> x * 2, naturals)   // nothing computed yet
    let first10  = DelayList.take(10, evens)             // forces exactly 10 elements
    println(first10)

take returns a plain List. At no point are more than 10 elements computed.

How the Lazy Chain Works

DelayList.from(1) does not generate an infinite list. It creates one node — head = 1, tail = lazy (from(2)). The tail is a recipe, not a value.

DCons(1, lazy ──► DCons(2, lazy ──► DCons(3, lazy ──► ...)))
              ↑                  ↑
          not yet computed   not yet computed

Each arrow is a deferred computation. take forces one node at a time, stopping when it has enough.

You can also filter lazily — only tested elements accumulate:

let odds = DelayList.filter(x -> x mod 2 != 0, DelayList.from(1))
println(DelayList.take(5, odds))   // 1 :: 3 :: 5 :: 7 :: 9 :: Nil

You describe a potentially unbounded computation. The runtime computes exactly as much as you ask for — no more.

The Reveal: Effect Handlers Were Delaying All Along

Recall the Ask handler from the previous chapter. Something important is happening at step 2:

def main(): Unit \ IO =
    run {
        greetUser()
    } with handler Ask {
        def ask(prompt, resume) =
            println(prompt);
            let answer = Console.readln();
            resume(answer)
    }

Execution flow

  1. greetUser calls Ask.ask(...)
  2. Execution PAUSES — the rest is suspended
  3. Handler prints the prompt
  4. Handler reads input — say, "Alice"
  5. Handler calls resume("Alice")
  6. greetUser RESUMES with name = "Alice"
  7. println("Hello, Alice!") runs

resume Is a Thunk

When Ask.ask(...) fires, the rest of greetUser() — every line after that call — is packaged up as a Unit -> a function and handed to the handler as resume. The handler holds the rest of the program. It can call it whenever it wants.

The three mechanisms are the same idea at different scales:

MechanismWhat is delayedWho holds the delay
Thunk () -> aA single expressionYou (the caller)
Lazy[T]A single pure expressionThe runtime
Effect handlerThe rest of the programThe handler

Effect handlers are the most powerful form — the delay covers not one expression but the entire remaining computation, all the way to the end of the run block.

Controlling the Delay

The handler receives resume — the suspended computation — and decides what to do with it:

Handler choiceEffectAnalogy
Call resume once, immediatelyNormal flow — program continuesOpen the envelope now
Never call resumeAbort — computation discardedShred the envelope
Call resume multiple timesFork — runs once per callPhotocopy and open each copy

The pattern is always the same. Only the handler's choice changes.

Pattern 1 — Immediate Resume

The most common pattern: handle the effect, then hand control back immediately.

eff Greet {
    def hello(name: String): Unit
}

def main(): Unit \ IO =
    run {
        Greet.hello("Alice");
        Greet.hello("Bob")
    } with handler Greet {
        def hello(name, resume) =
            println("Hello, ${name}!");
            resume()   // continue immediately
    }

The handler prints the greeting, then calls resume(). The program runs in a straight line. Both calls fire one after the other. Nothing surprising happens.

Pattern 2 — Never Resume (Abort)

A Void return type signals that the computation cannot resume. The handler receives _resume — but intentionally never calls it.

eff Fail {
    def fail(msg: String): Void
}

Void has no values — there is literally nothing to pass to a continuation. The computation cannot resume from that point.

def main(): Unit \ IO =
    run {
        println("Step 1");
        Fail.fail("something went wrong");
        println("Step 2")   // never runs
    } with handler Fail {
        def fail(msg, _resume) =
            println("Aborted: ${msg}")
    }

This is a typed, composable exception — no throw, no try, no catch. Just an effect whose handler chooses not to resume.

Pattern 3 — Call Resume Multiple Times (Fork)

eff Flip {
    def flip(): Bool
}

def main(): Unit \ IO =
    run {
        let a = Flip.flip();
        let b = Flip.flip();
        println("${a}, ${b}")
    } with handler Flip {
        def flip(_, resume) =
            resume(true);    // run rest with true
            resume(false)   // run rest with false
    }
// Prints: true, true
//         true, false
//         false, true
//         false, false

How it works

Calling resume multiple times does not loop. It runs the rest of the program multiple times from the same point — each time with a different value.

resume(true) runs the block with a = true. That hits the second flip, producing "true, true" and "true, false".

Then resume(false) runs the block again with a = false, producing the remaining two combinations.

This is how effect handlers express search, backtracking, and non-determinism — with no special syntax.

Real Problem 1: Infinite Sequences

You want the first five prime numbers. The naive approach — generate all primes first — either runs forever or requires guessing an upper bound. Lazy lists make the problem disappear:

def isPrime(n: Int32): Bool =
    n >= 2 and not List.exists(d -> n mod d == 0, List.range(2, n))

def main(): Unit \ IO =
    let nats   = DelayList.from(2)                 // 2, 3, 4, 5, ... forever
    let primes = DelayList.filter(isPrime, nats)    // filtered — still lazy
    let first5 = DelayList.take(5, primes)          // forces exactly 5
    println(first5)   // 2 :: 3 :: 5 :: 7 :: 11 :: Nil

nats is a description, not a list. primes is also a description. Nothing is computed until take(5, ...) forces exactly five elements — one at a time, stopping the moment five pass the test.

Real Problem 2: Retry Logic

Business logic that calls an external service — retried up to N times, without a single change to the business logic itself:

Business logic — unchanged

eff Fetch {
    def fetch(url: String): String
}

def reportWeather(url: String):
    Unit \ {Fetch, IO} =
    let data = Fetch.fetch(url);
    println("Weather: ${data}")

Retry handler — separate concern

def withRetry(n: Int32, action: Unit -> Unit \ {Fetch, IO}):
    Unit \ IO =
    if (n <= 0) println("Exhausted.")
    else
        run { action() }
        with handler Fetch {
            def fetch(url, _resume) =
                println("Retrying... ${n - 1} left");
                withRetry(n - 1, action)
        }

def main(): Unit \ IO =
    withRetry(3, () -> reportWeather("https://example.com/weather"))

The handler does not call _resume — it abandons the current attempt and starts fresh. Swap in a different handler and reportWeather never changes.

Summary

MechanismSyntaxEvaluates once?Can have effects?Power
Thunk() -> exprNoYesLow
Lazy[T]lazy (expr) / force xYesNo — pure onlyMedium
Effect handlerrun { } with handlerConfigurableYesHigh

Delay is a spectrum. Thunks are simple and universal — any language, any expression, no restrictions. Lazy[T] adds memoization and compiler-enforced purity for values that are expensive to compute and used in many places. Effect handlers are the full-power form: the suspended computation covers the entire rest of the program, and the handler can call it zero, once, or many times.

Flix gives you all three. The type system tells you which is in play at every point.
